# tests/validation

The characterization suite used in the master's thesis to measure how the
pipeline behaves: how long it takes, how reproducible it is, and whether the
epitopes it selects are backed by published literature. End users do not need
to run any of this; the everyday smoke check is `python tests/selftest.py`
(see the repository README), and anyone who just wants to watch the pipeline
run once should use the guided demo in the `main.py` menu (type `demo`). The
figures and tables here are what the thesis cites.

## What lives here

| File | Role |
|---|---|
| `seed_project.py` | Creates a disposable project (organisms, proteins, alleles, thresholds) non-interactively. |
| `run_replicate.py` | Runs one full pipeline replicate of a seeded project and records wall time per step. |
| `run_experiment_1.py` | Experiment 1 orchestrator (execution time + reproducibility). |
| `run_experiment_2.py` | Experiment 2 orchestrator (clinical literature validation via IEDB). |
| `collect_metrics.py` | Reads a finished replicate and extracts the per-stage epitope sets and counts. |
| `iedb_query.py` | Queries the IEDB IQ-API for a peptide: counts T-cell and MHC-I assays, collects PMIDs. |
| `make_charts.py` | Renders the publication PNGs from the raw per-replicate JSON. |
| `make_report.py` | Renders `figures/report.md` from the experiment summaries (do not hand-edit; regenerate). |
| `make_combined_table.py` | Renders `figures/exp_timings_wides_geral.xlsx`, one workbook with both experiments' numbers (timings + literature support). |
| `hardware_probe.py` | Captures host, package and tool versions so a run is reproducible. |
| `figures/` | Published PNGs + `report.md` (tracked in git). |
| `results/` | Raw per-replicate JSON (gitignored; regenerated by re-running an experiment). |

## Experiment 1: execution time and reproducibility

Walks four disposable projects, **HPV16_E7** and **SARS_NCAP** each at 1 track
and 3 tracks, 10 replicates each, strictly serial (one replicate finishes
before the next starts). It captures wall time per step and the per-stage
epitope sets, so the same track run repeatedly can be checked for determinism.
The pipeline's internal NetMHCpan + MHCFlurry threading inside `predict_binding`
is production-equivalent and left untouched; the suite never adds its own
parallelism.

Result: each track is deterministic across replicates (the same input gives the
same ★ set), with the only variation coming from the IEDB NetMHCpan endpoint
(about 6% of per-track runs return a transient anomaly, which the pipeline
detects rather than silently accepting).

```bash
# Full Experiment 1 (about 12 hours serial on a Ryzen 5 mobile)
python -m tests.validation.run_experiment_1

# Mini smoke (one project, one replicate)
python -m tests.validation.run_experiment_1 --projects hpv16_e7_tr1 --replicates 1
```

## Experiment 2: clinical literature validation via IEDB

Runs one multi-organism project (**HPV16_E7 + SARS_NCAP**, 2 tracks, `clinic`
preset) 10 times serially, then harvests two peptide sets per track:

- the **★ peptides** at the end of the pipeline, deduplicated across all 10 replicates;
- a **negative control**: the 10 peptides with the worst `max(net_percentile, flurry_percentile)` among those the consensus filter rejected at Stage 1 (the least binder-like peptides the pipeline threw out).

Each peptide in both sets is sent to the IEDB IQ-API to count T-cell and
MHC-Class-I assays and collect supporting PMIDs. The expectation: ★ peptides
recover known literature epitopes (for example SPRWYFYYL and LLMGTLGIV) while
the control set accumulates few or no hits.

```bash
# Full Experiment 2 (about 6 hours of pipeline + queries)
python -m tests.validation.run_experiment_2

# Mini smoke (one replicate)
python -m tests.validation.run_experiment_2 --replicates 1
```

## Regenerating figures and the report

After an experiment finishes (results in `results/exp1` and `results/exp2`):

```bash
python -m tests.validation.make_charts    # writes the PNGs into figures/
python -m tests.validation.make_report \
    --exp1-root tests/validation/results/exp1 \
    --exp2-root tests/validation/results/exp2 \
    --figures-root tests/validation/figures \
    --out tests/validation/figures/report.md

python -m tests.validation.make_combined_table   # writes figures/exp_timings_wides_geral.xlsx
```

Per preset, `make_charts.py` writes a funnel (`loss_by_stage`), the per-step
timing breakdown (`time_by_stage`), a total wall-time-per-replicate bar
(`time_total`, all steps stacked on a shared 0-280 s scale so 1-track and
3-track presets are comparable), a reproducibility heatmap (`consistency_matrix`)
and a consensus Venn (`venn`).

`results/` is gitignored, so a fresh clone has the figures and `report.md` but
not the raw per-replicate JSON. Re-run the experiments to recreate it.

## Reporting a failed run

`hardware_probe.py` captures the exact host and dependency versions:

```bash
python -m tests.validation.hardware_probe --out /tmp/hw.json
```

Attach `/tmp/hw.json`, the last lines of the failing run, and the exact command
to a GitHub issue. See the repository-root `TROUBLESHOOTING.md` for the common
install and connectivity problems before filing.
